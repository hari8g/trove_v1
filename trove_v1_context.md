## 1. What This Repository Does

Trove is an **AI-native code editor** built on VS Code Open Source Edition. Unlike an extension, it is a **first-class workbench contribution** compiled directly into the editor binary, granting it privileged file system, terminal, database, and HTTP access. Trove enables **multi-turn coding agents** that read, analyze, edit, and test code in real time, seamlessly integrated with VS Code's file explorer, editor, and terminal.

The core mission is to transform developer productivity by deploying an **agentic AI assistant** that understands your entire codebase, respects project constraints, and delivers inline code changes with full undo capability, approval workflows, and checkpoint-based recovery. Trove supports 15+ LLM providers (Anthropic Claude, OpenAI GPT-4, Google Gemini, Ollama, Groq, Mistral, etc.), with **built-in tools** for file I/O, terminal execution, code search (FTS5), web search (Tavily), and **Model Context Protocol (MCP)** plugins for extensibility.

**Target users:** individual developers and engineering teams who want AI-assisted coding without uploading code to external services (via self-hosted Ollama or on-premise endpoints), with granular control over agent behavior, token budgets, model selection, and workspace rules (via `.troverules` files).

**Key Architectural Decisions:**

1. **No code leaves the machine** — code only sent to LLM or web endpoints under explicit user control via API keys
2. **Agent loop is fault-tolerant** — retries failed LLM calls (rate limit, stream stall, truncation) with progressive context trim
3. **File edits are live & undoable** — apply to editor models immediately; checkpoints enable full rewind; user approval required
4. **Repo intelligence is incremental** — SQLite index updates only changed files (SHA-256 hash); symbols/chunks/summaries cached 24h
5. **IPC is synchronous where feasible** — tool execution blocks until completion; LLM calls stream async, post results via events
6. **Prompt caching reduces cost** — Anthropic native cache (1h TTL) for stable system, tools, conversation prefix; volatile state uncached; typical hit ratio 60–80% after turn 2

## 4. Module Wiring & Data Flow

### 4.1 Chat Loop (Core Agent)

**Entry Point:** User sends message in sidebar → `SidebarChat.tsx:onSendMessage()` → enqueues `QueuedUserMessage` in `chatThreadService.ts`

**Full Call Chain:**

1. `chatThreadService.ts:runQueuedUserMessages()` dequeues message
2. **Memory Intent** (optional, no LLM): `extractRememberIntent()` detects "Remember that …" → saves to `trove-memory.md` → returns early
3. **Plan Generation** (if enabled): lightweight LLM call → produces `PlanMessage` with 3–7 checklist items
4. **Checkpoint Creation**: `createCheckpoint()` snapshots all file content via `editCodeService.ts`
5. **Context Build**:
   - Stable system: `chat_systemMessage_stable()` (OS, mode, tools, repo, memory)
   - Volatile system: `chat_systemMessage_volatile()` (active file, terminals, directory tree)
   - Merge via `convertToLLMMessageService.ts`
   - Wire-level trim: `wireMessageTrim()` drops oldest tool bodies to fit budget
   - Apply prompt cache breakpoints
6. **Main Loop** (up to `maxAgentIterations` cycles):
   - **Send to LLM**: `sendLLMMessageService.ts` → IPC to main → `sendLLMMessage.impl.ts` → HTTP (with retries on 429/timeout/truncation)
   - **Parse Response**: extract tool calls, reasoning block, usage
   - **Deduplication**: `shouldSkipDuplicateFileRead()` skips already-read line ranges
   - **Parallel Read Batch** (if enabled): discover 4+ additional read-only tools → execute in parallel
   - **Tool Validation**: check params, check approval requirement
   - **User Approval**: for edit_file, rewrite_file, run_command, MCP tools → show modal → await click
   - **Tool Execution**: `toolsService.ts` → appropriate handler (read, edit, run, etc.)
   - **Append Results**: add `ToolResult` to conversation; mark plan items `done`
   - **Guardrail Injection**: append tail hints if patterns detected (repeat edit, repeat read, budget, etc.)
   - **Loop Break Conditions**:
     - Model stops calling tools (status: `ok`)
     - Reached `maxAgentIterations` (force-trim aggressively)
     - Reached `maxReadOnlyCalls` (skip read tools henceforth)
     - `maxConsecutiveToolFails` exceeded (stop)
     - Stream stall timeout
     - Output truncation detected (trim and continue 1 more turn)
7. **Finalize**:
   - Mark remaining plan items `done` or `skipped`
   - `agentDeliveryService.ts`: inspect terminal for build success / server start
   - Log token usage: `[Trove agent token usage] input:X cache_read:Y cache_write:Z output:W`
   - Store thread in VS Code storage

**Parameters In:** chatMessage (string), chatMode (Agent/Gather/Normal), selectedModel, fileSelections (URI[])
**Returns:** ChatMessage[], runTokenTotals
**Side Effects:** files get live diffs, terminals run, checkpoints saved, settings updated

### 4.2 Tool Execution

**Entry Point:** `toolsService.ts:executeBuiltinTool(toolName, params)`

**Read/Search Tools:**
- `read_file(path, startLine?, endLine?)` → file I/O (check dedup first)
- `get_file_outline(path)` → SQLite symbols query
- `get_symbol(path, symbolName)` → FTS lookup + extract line range
- `search_symbols(query)` → FTS5 BM25 ranking
- `search_codebase(query)` → FTS5 chunks table
- `search_for_files(pattern)` → ripgrep subprocess
- `search_pathnames_only(pattern)` → glob + filter
- `search_in_file(path, query)` → line-by-line scan
- `search_web(query)` → Tavily API
- `read_lint_errors(path)` → VS Code markers service
- `ls_dir(path)` → file I/O (cached per run)
- `get_dir_tree(path)` → recursive walk (cached per run)

**File Edits:**
- `edit_file(path, blocks)` → parse SEARCH/REPLACE blocks → apply via editCodeService → live editor update
- `rewrite_file(path, content)` → replace entire file → live editor update
- `create_file_or_folder(path)` → file I/O + editor notification
- `delete_file_or_folder(path)` → file I/O + editor notification

**Terminal:**
- `run_command(cmd, cwd?)` → one-shot child_process with stream capture
- `run_persistent_command(persistentTerminalId, cmd)` → write to PTY, await output
- `open_persistent_terminal(cwd?)` → node-pty allocation
- `kill_persistent_terminal(id)` → terminate PTY

**Returns:** `{ status: 'ok' | 'error' | 'invalid_params', content?, error? }`

### 4.3 Repo Intelligence & Symbol Index

**Entry Point:** Workspace opens → `repoIntelligenceService.ts` initializes

**Sequence:**
1. **Repo Scan**: detect languages, frameworks, build/test commands
2. **SQLite Startup**: load/create `<userData>/User/globalStorage/trove-repo-intelligence.db`
3. **File Metadata Load**: query `file_metadata` table; build SHA-256 hash map
4. **Incremental Refresh**:
   - Scan workspace files
   - For each file: compute SHA-256; compare to stored hash
   - Changed files: re-extract symbols, split into chunks
   - Insert/update `symbols`, `symbols_fts`, `code_chunks`, `chunks_fts` tables
5. **LLM Summaries** (one-time, 24h cache): projectPurpose, architectureSummary
6. **Load .troverules**: watch workspace roots for `.troverules` file; parse YAML → inject into system prompt

**Query Paths:**
- `search_codebase(query)` → FTS5 on `chunks_fts` with BM25 ranking
- `search_symbols(query)` → FTS5 on `symbols_fts`
- `get_file_outline(path)` → SQL SELECT from `symbols` WHERE file_path = ? ORDER BY line_range
- `get_symbol(path, symbolName)` → SQL lookup + extract source

**Side Effects:** SQLite writes (changed files only), repo profile injected into system prompt, `.troverules` merged into prompts

### 4.4 Message Pipeline & Prompt Caching

**Entry Point:** `convertToLLMMessageService.ts:convertChatToLLMMessages()`

**Sequence:**
1. **Stable System Block** (cached 1h on Anthropic):
   - OS info, mode rules, tool definitions (XML or JSON)
   - Repo profile, `.troverules`, user memory
   - File-reading policy (outline → symbol → range before full read)
2. **Volatile System Block** (uncached):
   - Active file (or outline if >3000 chars)
   - Open files list + outlines
   - Terminal IDs and status
   - Directory tree (cached per run)
3. **Conversation History**: truncate if exceeds context; drop oldest; keep last 2 user turns
4. **Tool Result Deduplication**: skip if same line range already read; append reference instead
5. **Wire Trim**: drop oldest tool bodies to fit character limit
6. **Prompt Cache Setup** (Anthropic native):
   - Breakpoint 1: stable system (high reuse)
   - Breakpoint 2: tools array (very high reuse)
   - Breakpoint 3: conversation prefix (all but last user message)
   - Volatile appended uncached
7. **Routed Claude**: wrap stable blocks in `<cache>` hints
8. **OpenAI**: use `prompt_cache_key: trove:${threadId}:${modelName}`

**Output:** `{ system: string | SystemBlock[], messages: LLMMessage[], cacheConfig? }`

### 4.6 LLM HTTP & Retry Logic

**Entry Point:** `sendLLMMessageService.ts:sendLLMMessage()` → IPC call

**Call Chain:**

1. Browser→Main IPC: `trove-channel-sendLLMMessage` → `sendLLMMessage.impl.ts`
2. **Provider Selection**: determine URL, headers, auth from settings + model
3. **Request Build**: messages, max output tokens, temperature, top_p
4. **HTTP Call**: native Node.js `https.request()` with streaming
5. **Retry Loop** (up to 3 attempts):
   - **Rate Limit (429)**: extract retry-after → wait → aggressive trim → retry
   - **Stream Stall**: exponential backoff → retry
   - **Truncation**: trim history aggressively → retry once
   - **Other Errors**: retry with fixed delay
6. **Response Parse**: extract tool calls, text, reasoning, usage
7. **Return to Browser**: post via IPC event

**Side Effects:** provider rate limit cooldown recorded, per-run token usage aggregated

- **Eager**: instantiate on startup
- **Delayed**: instantiate on first use

### Tool Definition Format
Tools declared in `common/prompt/prompts.ts`:
- **Anthropic native**: tool_use block (preferred)
- **XML fallback**: for models without native tool support
- Must be listed in `BuiltinToolCallParams` union

### Edit Tool Convention
SEARCH/REPLACE blocks follow strict format:
```
<<<<<<< ORIGINAL
// original code (MUST MATCH EXACTLY)
=======
// replacement code
>>>>>>> UPDATED
```
- Must match exactly (whitespace, comments, line breaks)
- Multiple blocks OK (each ORIGINAL must be disjoint)
- Preferred over `rewrite_file` for files >3KB

### IPC Channel Pattern
- **Browser→Main** (request): `serviceName.ts:method()` → IPC `trove-channel-xxx`
- **Main→Browser** (response): post event or register listener
- **Never block in renderer**: all main-process work is async (Promise or event)
- **Always include cancellation token** if long-running

### Approval Workflow
Destructive tools require user consent (unless `autoApprove.X` enabled):
- Call `toolsService.requestApproval(toolName, params)` before execution
- Modal appears; user clicks Approve/Reject
- Return `ToolResult` with status `ok` or `error`

### Symbol Extraction
- **Trigger**: file changes (SHA-256 hash mismatch)
- **Logic**: language-specific parser → symbols with line ranges
- **Storage**: `symbols` table + `symbols_fts` index
- **Query**: `search_symbols(query)` uses FTS5 BM25 ranking

### Checkpoint Pattern
Snapshots **all file state** at decision points:
- Before processing user message
- Before executing agent edits
- Stored in `CheckpointEntry[]` in thread storage
- Rewind: `chatThreadService.rewindToCheckpoint(checkpointId)`

### Agent Loop Guardrails
1. **Track state** during loop (e.g., edited files, read calls)
2. **Detect pattern** (e.g., same file edited twice)
3. **Inject tail hint** if triggered (appended before next LLM turn)
4. **Anthropic-safe**: hints always append after user role

### Compaction Logic
Tool results marked `compactable: true` (read/search):
- **Protected tail**: last 2 results + latest per file stay full
- **Stale bodies**: replaced with one-liner
- **Trigger**: after 4+ turns or exceeding budget

### Settings Schema Validation
All settings go through `validateSettings()`:
- TypeScript schema (not JSON schema) in `troveSettingsTypes.ts`
- Parse → validate → coerce types → store
- No partial updates; entire settings saved as one unit

### Memory Mechanism
1. **Trove Settings**: structured key-value store
2. **trove-memory.md**: plain text (user-visible, editable)
3. **Chat threads**: full conversation history in VS Code storage

### Read Tool Deduplication
Within a single agent run:
- Track all `read_file(path, start, end)` calls
- If later call requests line range **fully covered** by prior call → skip and reference instead
- Allows targeted range reads without full re-reads
